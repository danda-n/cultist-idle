Cultist Idle

UI/UX Design Specification — v0.18 · April 2026

01 — Aesthetic Direction

Forbidden Grimoire

The UI reads like a ritual manuscript — aged stone, blood-inked ledgers, candlelight pressing back the dark. Everything is deliberate, nothing decorative. Numbers feel weighed, not displayed. Actions feel committed, not clicked.

Tone reference: Darkest Dungeon × Cultist Simulator — grim folk ritual, not gothic camp. Candlelight. Chalk circles. Blood on stone.

Rule: if an element doesn't need to exist, it doesn't exist. Every pixel earns its place.

Design Principles

→ One screen. Everything the player needs is visible without navigating.

→ Negative states use colour, not popups. Red means fix this. Green means fine.

→ Progressive disclosure. Systems appear at their milestone, never before.

→ Atmospheric feedback. Animations reinforce the ritual theme — slow burns, not digital flashes.

→ Numbers are secondary. Direction (rising/falling) matters more than the exact value.

04 — Screen Layout

Three-column layout. The header bar is always visible. The left column holds systems that unlock progressively (Gateway, Expedition, Research, Constructs). The center column is the primary action area. The right column holds permanent progress context (Ritual Circle, Artifact slots, Activity log). No tabs required in normal play — all unlocked content is visible simultaneously via vertical scroll within each column.

Decision: Three columns rather than tabs. The player should see all active systems at a glance — a tab-based approach forces navigation for devotion monitoring. Columns allow passive awareness of everything unlocked.

Appears inline in the center column — not a modal overlay. The game world remains visible behind it. A horizontal banner slides in from the top of the center column with a brief animation. The player cannot send new expeditions while a Choice is pending but can still see all other systems.

08 — Implementation Notes for Developers

Stack decisions

React 19 + Zustand 5 + Tailwind CSS v4. Use CSS custom properties (this file's token system) as the design system foundation — Tailwind for layout utilities, custom props for colours and fonts.

Framer Motion for milestone card entrances and the Rehearsal fade transition. CSS-only animations everywhere else (fill bars, pulse effects, decay bars).

Responsive scope

Desktop-first. Minimum supported width: 900px. Below 900px, the layout collapses to single column with bottom tab bar (Left / Center / Right). The game is not intended for mobile portrait — tablets in landscape are the lower bound.

Mobile portrait is explicitly out of scope for Layer 1. Do not optimise for it.
Trifecta Triangle

Implement as inline SVG. Three <polygon> elements driven by resource state. fill is a CSS variable that transitions on threshold crossing. No canvas required.

Ritual Circle

SVG with 6 path segments. Crafted artifacts: stroke-dashoffset animates from full to 0 as resource % increases. Discovered: segment opacity transitions when expedition is active.

Fonts

Load via Google Fonts. Subset to Latin. Preconnect link in <head>. Fallback chain: 'Cinzel', serif — system serif is acceptable during font load.

09 — Accessibility (WCAG 2.1 AA)

The design targets WCAG 2.1 Level AA throughout. Dark atmospheric aesthetics and accessibility are not in conflict — the palette is intentionally built so that all text tokens pass contrast requirements on all background tokens in this system.

Rule: Every colour token used for readable text must achieve ≥ 4.5:1 contrast on its intended background (WCAG 1.4.3 — normal text). For large text ≥ 18pt (24px) or 14pt bold, the threshold is 3:1. SVG fills, progress bars, borders, and indicator dots are governed by WCAG 1.4.11 (non-text contrast, ≥ 3:1) — this is why --indicator-* tokens exist separately from --state-*-text.

Implementation Rules

Never do this

→ Use opacity on text elements. Opacity multiplies against the colour's contrast ratio, producing effective ratios that are hard to audit and always lower than intended. Use an explicit colour token instead.

→ Use any --indicator-* token as a text colour. These are indicator fills (3:1 non-text minimum) — they do not meet the 4.5:1 text requirement.

→ Use --text-dim or --text-ghost for any information the player needs to read. Reserve them for empty-slot placeholders and ornamental text in empty UI states.

→ Use text smaller than 11px (0.7rem) for any information that carries meaning. Decorative-only exceptions apply only to ornamental characters.

Do this instead

→ Use --text-secondary for any secondary or de-emphasised text that still carries information (rates, timers, labels, hints). 6.3:1 on all dark backgrounds.

→ Use --state-green-text, --state-warn-text, --state-danger-text for status labels, badges, and category headers that convey a state.

→ Pair --indicator-* fills with a text-safe token for any label that sits on top of or alongside the indicator.

→ For disabled/locked UI: use opacity: 0.35 on the entire element (not just the text), so that the visual dimming applies uniformly and the locked state is communicated by more than colour alone.
